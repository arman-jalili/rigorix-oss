## Business Context

**Rigorix** is a template-driven DAG execution engine with bounded autonomy. It takes a user's natural-language intent (e.g., "add a new API endpoint", "refactor the auth module"), classifies it against predefined TOML workflow templates, generates an executable Directed Acyclic Graph (DAG) plan, executes the plan in parallel with configurable retry/fallback logic, enforces safety limits (resource budgets, risk gating), and streams observability events for real-time monitoring.

The **cli** module is the forthcoming user-facing CLI that wraps the `rigorix-engine` library. It provides:
- A terminal UI (TUI) for interactive execution monitoring
- Command-line command dispatch (run, plan, init, etc.)
- Configuration file loading and validation
- Execution session management (view history, replay audit logs)
- Integration with the Guardian agent framework for AI-assisted development

The engine (`engine/` crate) is already implemented with 17 bounded contexts following Clean Architecture (domain/application/infrastructure/interfaces). The CLI module will consume the engine as a library and surface its capabilities through a user-friendly command-line interface.

**Core principles:**
1. **Template-driven** — Workflows defined in TOML templates, not dynamically generated by LLM
2. **DAG-based** — Tasks as nodes with dependency edges, executed in topological order
3. **Minimal LLM** — LLM is a planning tool only, not used at execution time
4. **Bounded autonomy** — Hard caps on dynamic behavior (max retries, max tokens, max tool calls)
5. **Risk-gated** — Safe operations auto-execute, medium operations require confirmation, dangerous operations run in dry-run mode
6. **Pre-validated** — Validators catch errors BEFORE execution
7. **Auditable** — Every planning decision is tracked, hashed, and diffable

## Assumptions

| Assumption | Impact if Wrong | Mitigation |
|------------|----------------|-----------|
| LLM API is available and responsive during planning phase | Planning fails; user gets "LLM unavailable" error with fallback to cached templates | Implement offline mode with pre-compiled templates; configurable planning timeout |
| User has write access to the project directory | State persistence and tool execution fail with permission errors | Validate directory permissions at startup; show actionable error messages |
| TOML templates are well-formed and validated before execution | Invalid templates cause execution to start with broken DAG | Pre-flight validation at `plan` step; reject invalid templates at load time |
| User's terminal supports Unicode and ANSI escape sequences (TUI) | TUI renders garbled text; falls back to plain text output | Detect terminal capabilities at startup; provide `--no-tui` flag for plain output |
| Engine library APIs are stable (contract-freeze respected) | CLI breaks when engine interfaces change | Pin engine version in Cargo.toml; run CI with latest engine commit; use semver |
| Concurrent tool executions do not interfere with each other | Race conditions in file system operations corrupt state | Engine enforces file-level locking; tools should be idempotent where possible |
| User intent is expressible within the existing template catalog | No matching template found; fallback to LLM-based template generation may fail | Template generation module provides fallback; user is prompted to refine intent |
| The file system has sufficient space for state persistence | State writes fail; execution audit data is lost | Pre-flight disk space check; graceful error handling with user notification |
| Network access is available for LLM calls and remote audit backends | Planning fails; audit delivery is deferred to local queue | Offline queue for audit envelopes; configurable retry with circuit breaker |

### Detailed: Plugin System (Q2) — Recommendation: Defer to v2

**Why not now:**
- Rust has no stable ABI, so dynamic loading (`dlopen`/`libloading`) is fragile and unsafe
- WASM-based plugins would require embedding a WASM runtime (increased binary size, complexity)
- The engine already has a `Tool` trait — but plugging a new Rust trait implementation requires recompiling the binary anyway

**What to do instead for v1:**
- The `RunCommand` action in `TemplateAction` already lets users execute arbitrary shell commands
- Users can define custom tool wrappers as shell scripts and reference them via `RunCommand`
- In `rigorix.toml`, add a `[[tool_aliases]]` section to allowlist specific commands with risk levels:
  ```toml
  [[tool_aliases]]
  name = "my-lint"
  command = "./scripts/lint.sh"
  risk_level = "low"
  ```
- This covers ~90% of plugin use cases without any dynamic loading

**v2 possibility:** WASM-based plugin system where users write tools as WASM modules with a well-defined ABI, loaded at runtime. This is a significant feature — size it separately.

### Detailed: Daemon Mode (Q4) — Recommendation: Ephemeral CLI for v1

**Why not now:**
- Daemon mode adds massive complexity: process lifecycle management, Unix sockets / IPC, client-server protocol, health checks, graceful restart
- The engine already has crash-safe atomic state persistence — restarting an ephemeral CLI after a crash recovers state without a daemon
- Debugging is harder (two processes, log correlation)

**What to do instead for v1:**
- Ephemeral CLI: detect leftover state files from a crashed run on startup, offer to resume or discard
- Fast startup via config caching (mtime-based invalidation of parsed `rigorix.toml`)
- The EventBus holds events in memory during a run and persists at end — no daemon needed for audit trails

**v2 possibility:** A `rigorix daemon` for watch mode (`rigorix watch` re-runs plan on file changes) or IDE integration (LSP-like protocol). Size separately.

### Resolved
- **Template Generation UX**: Added FR-019 through FR-023. The `rigorix generate` command provides explicit template creation with optional `--dry-run` and `--stdout` flags. The automatic fallback during planning also persists generated templates. Flow: `generate` → persist → `plan`/`run`, or `run` → no match → auto-generate → persist → execute.
- **TUI**: ratatui confirmed.
- **Plugin System**: Use configurable RunCommand actions for v1, WASM plugins deferred to v2.
- **Multi-Repo**: Single repo for v1. Simple path-based expansion later.
- **Daemon Mode**: Ephemeral CLI for v1. Daemon deferred to v2.
- **Migration Docs**: Required. Will create `docs/MIGRATION.md` post-scaffold.

## Aggregate Roots

| Aggregate Root | Bounded Context | Key Entities | Invariants |
|---------------|----------------|--------------|------------|
| **Config** | Configuration | Config, Secret, EnforcementPreset | Config must be valid after merging; secrets never serialized to plaintext output |
| **Template** | Templates | Template, TemplateNode, ParameterDef | TOML must parse; all node references must resolve; no cycles in template graph |
| **TaskGraph** | DAG Engine | TaskGraph, TaskNode, ExecutionPolicy | No duplicate node IDs; all dependency references valid; no cycles after seal(); sealed → immutable |
| **ExecutionResult** | Execution Engine | ExecutionResult, TaskResult, NodeExecutionState | Each node has exactly one terminal TaskResult; retry count ≤ max_attempts |
| **ToolRegistry** | Tool System | Tool, ToolRegistry | No duplicate tool names; all required tools registered before execution starts |
| **ExecutionEnforcer** | Enforcement | ExecutionEnforcer, EnforcementConfig, ResourceBudget | Budget used ≤ budget limit; tool policy always evaluated before execution |
| **RiskConfig** | Risk Gating | RiskConfig, RiskClassifier | Each tool pattern maps to exactly one RiskLevel; default risk for unmapped tools |
| **LlmBudget** | Budget Tracking | LlmBudget, LlmBudgetReservation | Reserved + used ≤ total budget; reservations auto-return on Drop |
| **EventBus** | Event System | EventBus, PersistedEvent | Events are append-only; sequence numbers are monotonic; no event loss in memory |
| **AuditEnvelope** | Audit | AuditEnvelope, AuditSender | Envelope contains valid execution_id; events are ordered by sequence; hash chain integrity |
| **ExecutionState** | State Persistence | ExecutionState, NodeState, ExecutionGraph | Atomic write-rename guarantees atomicity; state file must be valid JSON |
| **SymbolGraph** | Repo Engine | SymbolGraph, SymbolDefinition | Symbol names are unique within a file; O(1) name lookup; cross-references are resolved |
| **CliConfig** | CLI Boundary | CliConfig, CliCommand | CLI flags override config file values; config file overrides defaults; output format is valid (human/json) |
